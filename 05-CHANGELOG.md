## Истины и инварианты (НЕ забывать, проверять перед правкой)
- **SCO-путь НЕ удалён.** Вопреки старым заметкам. Он условен от `btScoMicAvailable()` и поднимается
  через `applyMicRouting` (`MODE_IN_COMMUNICATION` + `setCommunicationDevice`).
- **Магнитола LADA Vesta = HFP-устройство.** Поэтому в авто `wantHeadsetMic()` → true → ложный
  «вызов» на магнитоле + провал короткой диктовки на встроенный микрофон. Наушники без музыки —
  чистый кейс; авто — проблемный.
- **`wantHeadsetMic()` с 8.12 привязан к ОКНУ ЗАХВАТА, не к `state==AWAKE`.** Условие теперь
  `dictation || aiListening || aiDialog || recordingVoice`. До 8.12 SCO висел всю AWAKE-сессию →
  магнитола держала ложный «вызов» десятками секунд (в логе 15:05:29→15:07:07, ~98 c). `aiDialog`
  в условии держит SCO между ходами «поговорим» — без мигания канала на каждой реплике.
- **`goToSleep()` до 8.12 НЕ отпускал SCO** — маршрутизация сбрасывалась только через `idleRunnable`.
  С 8.12 `goToSleep()` зовёт `refreshMicForState()`. При правках сна следить, чтобы SCO отпускался.
- **Голосовое в тишину — ДВЕ разные причины, не путать (уточнено 8.13):**
  - (а) *Смена аудиорежима посреди записи.* Если под запись приложение входит в `MODE_IN_COMMUNICATION`
    (поднимает SCO), Telegram'овская запись рвётся за ~0.5 c → уходит пустое голосовое, плюс на
    магнитоле зажигается ложный «вызов». Это случилось в 8.12, когда `recordingVoice` попал в
    `wantHeadsetMic`. Доказательство: SCO был на гарнитуре (встроенный свободен!), а запись всё равно
    сорвалась — значит ломает САМ переход режима, а не контест микрофона. С 8.13 под запись
    аудиорежим НЕ трогаем (NORMAL), `recordingVoice` убран из `wantHeadsetMic`.
  - (б) *Контест за один встроенный мик* (без BT, наушники A2DP): приложение и мессенджер на одном
    встроенном → мессенджер получает тишину. Это отдельный, пока открытый случай (#2b), чинится не
    раньше замера FreeBuds SCO.
  - **Вывод-инвариант:** под запись голосового аудиорежим оставлять NORMAL и SCO не поднимать. В
    машине запись работает именно в NORMAL (встроенный свободен, режим не дёргается).
- **Облачный ИИ: `readTimeout` — таймаут НА ОДНО чтение, не на весь ответ.** На плохой сети ответ
  тёк по байтам, каждое чтение укладывалось в 13 c, и «думаю» висело минутами (в логе ~5 мин) вместо
  отката на локальную. С 8.13 облачный вызов обёрнут общим wall-clock дедлайном ~16 c
  (`LocalAi.withTimeout`) → быстрый откат на офлайн-модель. При полностью выключенном интернете
  локальная срабатывает сразу (cloud пропускается по `Net.isOnline`).
- **Во время «думания» (`aiThinking`) микрофон не нужен → отпускать SCO.** Иначе ложный «вызов» висел
  всю генерацию (минуты на плохой сети). С 8.13 `handleAi` зовёт `refreshMicForState()`. В диалоге
  «поговорим» (`aiDialog`) канал удерживается — wantHeadsetMic его держит между ходами.
- **Голосовое в тишину = конфликт за ОДИН физический микрофон, а не «запись ломает звук».**
  Мессенджер всегда пишет со встроенного. Если приложение в этот момент тоже на встроенном (нет
  BT-микрофона) → один мик на двоих → мессенджер получает тишину. В машине приложение было на SCO →
  встроенный свободен → голосовое записалось нормально. Вывод: под запись голосового приложение и
  мессенджер не должны быть на одном мике (либо приложение на BT-мике, либо отпускает встроенный).
- **A2DP (музыка) и SCO (микрофон) по Bluetooth взаимоисключающи.** Нельзя одновременно слушать
  музыку через BT и писать с BT-микрофона без разрыва звука.
- **Музыку под команду на этой магнитоле (Vesta) глушим ПАУЗОЙ плеера (уточнено 8.19), НЕ
  аудиофокусом и НЕ setStreamVolume.** Аудиофокус MAY_DUCK плеер не возвращает (залипает тихим, 8.17),
  setStreamVolume дрейфует по A2DP (8.14). Единственный рабочий рычаг — пауза/плей
  (`dispatchMediaKeyEvent`), который плеер слушается. Плей делать ТОЛЬКО если паузу ставили мы
  (`musicPausedByUs`), плюс страховочный авто-возврат по таймауту. Пауза/плей глобальны (бьют по
  активной медиа-сессии) — для авто с одним плеером норм.
- **Порядок веток `onResult` — инвариант.** Антидубль → глубокий сон → `isSpeaking` → `aiThinking`
  → `recordingVoice` → `aiListening` → звонок → экран → медиа → диктовка → wake → команда.
  Отмена раньше создания; tomorrow/free/urgent раньше общих триггеров.
- **Анти-эхо:** `isSpeaking` нельзя снимать мгновенно — держать грейс (`speechTailGraceMs`, ~1300 мс)
  и ждать реального конца TTS (`tts?.isSpeaking`), иначе распознаватель ловит своё эхо. Барж-ин
  «иван стоп/подробнее» сохранять.
- **Отложенное действие после речи (`afterSpeak`, ветки `speakThen`/`onDone`) с 8.12 идёт через
  `fireAfterSpeak()`** — тот же анти-эхо-гейт, что `resumeAfterSpeak` (ре-проверка `tts?.isSpeaking`
  с репостом каждые 400 мс). Без этого ре-арм хода «поговорим» ловил хвост собственной озвучки.
- **Новые триггер-слова бесполезны без `Vocabulary.kt`** (грамматика Vosk).
- **Сборка:** новые файлы — `git add -A` (не `commit -am`). Версия — в трёх местах. Zip — полное дерево.
- **`val`, ссылающийся на себя в инициализаторе → ошибка компиляции.** Использовать
  `object : Runnable { override fun run() { ... } }` с `this`.
- **Память секретаря** (`secretary.json`) — проекты/люди учатся из речи, никогда не хардкодить.
- **Захват СВОБОДНОЙ речи (вопрос к ИИ, имя, текст для «сформулируй») — через облачный Whisper
  (`CloudStt`/`startCloudCapture`), НЕ через Vosk (уточнено 8.18).** Vosk работает по грамматике
  (`Vocabulary.kt`) и глушит любое слово вне словаря — редкие/медицинские слова («дефолаг»→«блюз фолк»)
  не распознаются. Vosk — ТОЛЬКО для команд. Whisper можно завязывать на Vosk лишь как ОТКАТ при
  отсутствии сети, не как основной путь.

### 8.14 — приглушение музыки под звонок и приём команд
Затронут `VoiceRecognitionService.kt`. Новых файлов нет.
- **Проблема (из надиктованных пояснений в поездке):** на входящем музыка из колонок авто (A2DP) не
  приглушается → микрофон телефона не слышит «иван ответь» из-за маскировки, приходится кричать, всё
  равно не ловит. То же — при обычном приёме команд поверх музыки.
- **Причинная модель:** динамики авто далеко от мика телефона, эхоподавление (AEC) не вытягивает →
  голос тонет в музыке. Лечится понижением громкости `STREAM_MUSIC` на время, когда реально слушаем.
- **Сделано:** `updateMusicDuck()` + `wantMusicDuck()` = `callActive || state==AWAKE || dictation ||
  aiListening || aiDialog`. Сохраняем текущую громкость → понижаем до 15% (но только если выше) →
  возвращаем сохранённую, когда слушать перестали. Вызовы: в `onResult` (после `handleResult` —
  синхронизация с новым state), в `refreshMicForState` (восстановление при засыпании/смене режима),
  в `phoneListener` (на входящем — мгновенно). В ASLEEP (только вейк-слово) музыку НЕ трогаем —
  иначе была бы постоянно тихой. Логи: «Музыка приглушена для приёма команд (…→…)» / «восстановлена».
- Риск/проверить на устройстве: понижение `STREAM_MUSIC` со стороны телефона может НЕ управлять
  громкостью магнитолы/CarPlay (зависит от того, как авто принимает A2DP). По логу «приглушена/
  восстановлена» + по слуху проверить, реально ли стало тише и слышит ли «иван ответь». Если магнитола
  не слушается `STREAM_MUSIC` — приглушать вслепую дальше НЕ будем, искать другой рычаг.
- Что не трогали: медиа-режим (просмотр видео/музыки с «иван пауза») — там музыку дёргать на каждую
  команду хуже; если понадобится — отдельно. Громкость звонка (`STREAM_VOICE_CALL`) и рингтона
  (`STREAM_RING`) не трогаем — глушим только музыку.

### 8.15 — приглушение музыки через аудиофокус (фикс «громкость не возвращается»)
Затронут `VoiceRecognitionService.kt`. Новых файлов нет.
- **Регресс 8.14:** приглушение через `setStreamVolume(STREAM_MUSIC)` понижало громкость, но НЕ
  возвращало. Две причины (по логу за сутки): (1) **перезапуск сервиса** ОС среди дня (`[SVC] Служба
  запущена`) обнулял поля `musicDucked`/`savedMusicVol` → музыка оставалась на 2 навсегда; (2) **дрейф
  абсолютной громкости A2DP** — магнитола/CarPlay не принимает обратный `setStreamVolume` надёжно: в
  логе пишем «восстановлена →15», а по факту на устройстве осталось 2, и следующее приглушение
  сохраняет уже 2. База весь день съезжала 15→9→5→4→3→2.
- **Сделано (8.15):** механизм заменён на **аудиофокус** `AUDIOFOCUS_GAIN_TRANSIENT_MAY_DUCK`
  (USAGE_ASSISTANT). `want=true` → `requestAudioFocus` (плеер сам притишивается); `want=false` →
  `abandonAudioFocusRequest` (плеер сам возвращает громкость). Поле `savedMusicVol` удалено, добавлено
  `duckFocusRequest`. `onDestroy` снимает фокус. Логи: «Музыка приглушена (аудиофокус MAY_DUCK…)» /
  «восстановлена (аудиофокус отпущен)». Условие `wantMusicDuck()` не менялось.
- **Почему надёжно:** фокус не трогает абсолютную громкость (нет дрейфа A2DP) и переживает
  перезапуск сервиса — если процесс умирает, ОС снимает фокус автоматически → плеер восстанавливает
  громкость сам. «Сон забирает громкость» исчезает: уход в сон = отпускание фокуса = возврат музыки.
- Риск/проверить: глубина приглушения теперь задаётся плеером (обычно ~20%), а не нашими 15% — если
  на машине окажется недостаточно тихо для мика, поднимать отдельно (не возвращаться к setStreamVolume
  — он дрейфует). Если плеер/магнитола игнорируют MAY_DUCK — музыка не притишится, НО и не застрянет
  (строго безопаснее 8.14). Проверить по слуху: тише ли при «иван», слышно ли «иван ответь», и
  ВОЗВРАЩАЕТСЯ ли громкость после сна/звонка.

### 8.16 — дебаунс отпускания фокуса (фикс «плеер залип тихим»)
Затронут `VoiceRecognitionService.kt`. Новых файлов нет.
- **Симптом 8.15 (по логу + слуху):** иногда музыка оставалась очень тихой при максимальной громкости,
  лечилось паузой/плеем. Парность фокуса идеальна (9/9 приглушено/восстановлено), сервис не
  перезапускался — значит фокус мы отпускаем, но САМ ПЛЕЕР не поднимал себя обратно.
- **Корень:** дёрганье фокуса. В логе циклы «отпустили → через 1–2 c снова приглушили»
  (16:38:49→16:38:50, 16:44:34→16:44:36). Плеер не успевал доехать до полной громкости по рампе —
  новый запрос фокуса застаёт его на середине, и он залипает тихим. Пауза/плей сбрасывают внутреннее
  состояние плеера.
- **Сделано (8.16):** отпускание фокуса через ДЕБАУНС — `releaseDuckRunnable` с задержкой 3000 мс.
  `want=true` отменяет отложенное отпускание и держит фокус непрерывно весь активный диалог;
  `want=false` планирует отпускание через 3 c (короткий «сон→снова команда» фокус не дёргает).

  мог взять эту Vosk-кашу как суть плана.
- **Три корня и три фикса:**
  - **(1) Плохое распознавание = Vosk-каша в плане.** ОТКАТ инлайн-разбора 8.26: `startPlanQuery` теперь
    ВСЕГДА пишет через `startCloudCapture` (Whisper), `inlineText` игнорируется. Vosk коверкает свободную
    речь — в план его пускать нельзя. Суть — только Whisper (онлайн) или Vosk-офлайн по тому же звуку
    (когда нет сети). `afterTrigger` удалён.
  - **(2) Наложение «слушаю» на «слушаю» = гонка захватов.** ЗАМОК: в начале `startCloudCapture`
    `if (cloudCapturing) { лог; return }` — пока идёт захват, второй НЕ стартует. Два AudioRecord на один
    микрофон физически исключены. Все пути захвата (ai_ask/plan/memory/переспрос) идут через единый
    `startCloudCapture` → замок покрывает всё.
  - **(3) Отключение микрофона = замок мог залипнуть.** Тело записи обёрнуто в `try/catch`; `post{}` со
    сбросом `cloudCapturing=false` выполняется ВСЕГДА (успех/пусто/исключение). Если бы запись бросила
    исключение — замок остался бы `true` навсегда, захват залип бы. Теперь замок снимается гарантированно,
    затем `returnToListening()` (безусловный возврат Vosk).
- Риск/проверить на устройстве: (1) «иван напомни посчитать зарплату сегодня в 9 вечера» → в заголовке
  ТОЧНЫЙ Whisper-текст, не «сорок прочитай». (2) Быстрые повторные «иван напомни» подряд → второй
  отклоняется («Захват уже идёт»), нет наложения. (3) Любой сбой записи → микрофон возвращается, не виснет.
- **Истина-инвариант:** суть плана/вопроса — ТОЛЬКО Whisper (или Vosk-офлайн по тому же буферу при
  отсутствии сети), НИКОГДА не Vosk-строка пробуждения (она грамматическая, коверкает). Захват под
  ЗАМКОМ `cloudCapturing` — повторный старт отклоняется. Замок снимать в `post{}` ВСЕГДА (try/catch),
  иначе залипнет. 429 от OpenRouter (LLM-разбор) НЕ ломает создание планов — есть `localParse`; Whisper
  (STT, отдельный сервис Groq) от этого не страдает.

## Открытые причинные узлы (куда копать дальше)
- **Голосовое пишется в тишину без BT-микрофона (новый, P1/P2).** Корень — конфликт за встроенный мик
  приложения и мессенджера. В машине (SCO) работает, в наушниках (A2DP, SCO не встал) — тишина.
  План: под запись голосового либо поднимать SCO на FreeBuds (HFP-мик есть, экрана нет → ложного
  «вызова» не будет), либо отпускать встроенный. НО: в логе нет подтверждения, что на FreeBuds SCO
  реально встаёт — нужен замер (надеть FreeBuds, «иван напиши», проверить `scoOn=true` с FreeBuds).
  Чинить только после замера (анти-паттерн: аудио вслепую).
- **Диалог «поговорим» — защита от окружения (ТВ/дети).** Анти-эхо на ре-арм закрыт в 8.12, но в шуме
  чужая речь всё ещё может стать «вопросом». Осталось: окно приёма после ответа / подтверждение на
  продолжение / авто-выход по тишине или N мусорных реплик.
- **Аудио в авто:** нормальный BT-микрофон по замерам 8.11 (ждать CONNECTED, удержание канала на
  сессию захвата, приглушение музыки, политика наушники/авто, усиление встроенного). Не вслепую.
- **Планирование офлайн:** `Secretary.plan` целиком на облаке → нужен внятный офлайн-ответ.
